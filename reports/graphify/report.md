# Отчёт по проекту MiroVizor — graphify-анализ

Дата: автогенерация. Источник: `reports/graphify/`.

---

## 1. Что было сделано

1. **Запущен graphify** — vendored-движок построения интерактивных графов (tree-sitter + NetworkX + vis.js).
2. **Починен `miro_vizor/graphify_adapter.py`** — прежний адаптер вызывал несуществующие функции vendored graphify (`generate_code_graph`, `generate_text_graph`, `visualize_graph`). Реальный API: `collect_files`, `extract`, `build_from_json`, `cluster`, `to_html`. Адаптер переписан под реальный API, сохранена обратная совместимость (`build_levels_graph`, `export_graph_html`).
3. **Построены графы:**
   - `code_graph.html` — граф всего кода `miro_vizor` (включая vendor): 821 узел, 1258 рёбер, 52 сообщества.
   - `code_graph_own.html` — граф **только собственного кода** `miro_vizor` (без vendor/tests): 101 узел, 180 рёбер, 10 сообществ.
   - `text_graph.html` — граф уровней из `sample.txt`: 20 узлов, 21 ребро, 6 сообществ.
4. **Размечен `sample.txt`** — `result.json`, `result.html`, `result.csv`.
5. **Запущены тесты** — 20/20 базовых тестов проходят.

### Архитектурные выводы по коду

1. **Чёткая слоистая архитектура.** Граф выделяет 3 слоя:
   - **ядро** (`ontology.py`, `scorer.py`, `tokenizer.py`, `marker.py`, `embedder.py`, `languages.py`) — онтология + скоринг + токенизация;
   - **экспорт** (`exporter.py`, `graphify_adapter.py`) — JSON/CSV/HTML + графы;
   - **интерфейсы** (`cli.py`, `app_api.py`, `app_streamlit.py`, `app_admin.py`) — CLI + FastAPI + Streamlit.

2. **`MiroOntology` — центральный узел ядра** (степень 12). Онтология используется всеми слоями, что ожидаемо для предметно-ориентированного инструмента. Это не god-объект: класс инкапсулирует только загрузку/индекс ключевых слов.

3. **`app_api.py` — самый связный модуль** (степень 15). Концентрирует endpoint-функции и `_get_marker()` — фабрику маркера. Это точка интеграции ядра с веб-слоем. При росте числа endpoint-ов стоит рассмотреть вынос graph-эндпоинтов в отдельный router.

4. **`graphify_adapter.py` — мост к vendored graphify** (степень 12 после починки). Корректно изолирует детали vendored-библиотеки. **Найденный дефект:** ранее адаптер был полностью сломан — вызывал несуществующие функции; CLI `--graph-code` / `--graph-text` падал. Теперь исправлен.

5. **Нет циклических зависимостей** между модулями ядра — граф направленный, `imports_from` образуют ацикличный поток: interfaces → marker → (ontology + scorer + tokenizer + embedder).

6. **13 файлов собственного кода** — компактный, читаемый проект. Vendor (graphify) добавляет ~720 узлов шума в полный граф, поэтому собственный код анализировался отдельно.

## 3. Анализ текста sample.txt — спектр Миросложения

**Доминантный уровень труда:** **L3** (Оранжевый — Цель/Действие).

**Спектр по уровням:**
| уровень | доля | цвет | тема |
|---------|------|------|------|
| L1 | 0.000 | Красный | Воля |
| L2 | 0.167 | Жёлтый | Расцвет |
| L3 | 0.250 | Оранжевый | Цель |
| L4 | 0.250 | Зелёный | Гармония |
| L5 | 0.167 | Голубой | Мастерство |
| L6 | 0.167 | Синий | Идеи |
| L7 | 0.000 | Фиолетовый | Духовное |

**Предложения с определённым доминантным уровнем:**
- «Мир изменился, вижу целью Прибавить мерности ему.» → L4/L3 (Мир → L4, целью → L3)
- «Моя душа звенит капелью, Идеи смыслов на кону.» → L6 (Идеи)
- «Посему Мы изменились, вы в расцвете…» → L2 (расцвете)
- «Орлицей смелой обернулись, К высоким целям упорхнув.» → L3 (целям)
- «Струной поющей прикоснулись, Мне в сердце музыку вернув.» → L4 (сердце)
- «Нам всем порой нужна отвага…» → L5 (отвага)

**Граф уровней текста:** 20 узлов, 21 ребро, 6 сообществ. Структура: Труд → Глава → 12 предложений → 4 уровня (L2, L3, L4, L6). L3 и L4 доминируют совместно.

### Выводы по тексту

1. **Текст балансирует между действием (L3) и гармонией (L4)** — это лирико-целевое стихотворение: образы полёта/цели переплетаются с образами сердца/мира.
2. **L1 (Воля) и L7 (Духовное) отсутствуют** — текст не опирается на прямые маркеры воли или духовности; онтология не содержит синонимов для «время», «лекарь», «успех», «мастер», «судьба», «да».
3. **Покрытие онтологии:** 7 из 12 предложений (~58%) получили непустую разметку. 5 предложений не размечены — их лексика («Предвижу всё», «дивное волненье», «Три года срок», «Приму ответ», «Успех во всём») не представлена в `miro_ontology.json`.
4. **Рекомендация по онтологии:** добавить синонимы/ключевники для слов: *время, лекарь, успех, мастер, судьба, отвага (уже есть), ответ, да, цветок, срок* — чтобы поднять покрытие до 80–90%.

| `graphify_adapter.py` | ✅ исправлен — был сломан, вызывал несуществующие функции |

### Найденные и устранённые дефекты

1. **`graphify_adapter.py` — критический:** импортировал `generate_code_graph`, `generate_text_graph`, `visualize_graph` из `graphify`, которых в vendored версии не существует (API: `collect_files`/`extract`/`build_from_json`/`cluster`/`to_html`). Все CLI/API вызовы `--graph-code` и `--graph-text` падали с `RuntimeError: graphify is not available`. **Исправлено:** адаптер переписан под реальный API, функции импортируются напрямую из подмодулей (обход конфликта имя-подмодуля/имя-функции в lazy `__getattr__`).

### Рекомендации

1. **Покрытие онтологии** — расширить `miro_ontology.json` синонимами для пропущенной лексики sample.txt (см. §3 п.4).
2. **`app_api.py`** — при росте endpoint-ов вынести `/graph/*` в отдельный `APIRouter`.
3. **Граф кода** — по умолчанию строить граф только собственного кода (исключать `vendor/`), либо добавить CLI-флаг `--graph-include-vendor`.
4. **Тесты graph-API** — изолировать от тяжёлых зависимостей (torch/sentence-transformers); использовать `use_embeddings=false` и моки, чтобы тесты не падали в окружениях без GPU/с проблемным torch.
